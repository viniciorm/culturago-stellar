Desplegando CulturaGO en un VPS de 4GB con Agentes de IA

De Next.js 16 y Supabase Self-Hosted a producción en Linux Ubuntu utilizando pnpm, Docker y asistencia multi-agente de Inteligencia Artificial.

✍️ Por Vinicio & Antigravity AI 📅 Julio 2026 🚀 FDVC 2026 MVP

Resumen Ejecutivo: Esta guía documenta la experiencia completa de arquitectura y despliegue del MVP para CulturaGO (Festival Nacional Danza del Vientre Chile 2026) en un VPS Ubuntu de 4GB RAM en Truebox. Cubre la migración del motor de paquetes a pnpm, la resolución de errores de Docker/TypeScript, la configuración de Supabase Self-Hosted y la interacción de desarrollo colaborativo con agentes de Inteligencia Artificial (Hermes y Antigravity).

1. Arquitectura del Sistema y Elección de Infraestructura

CulturaGO es una plataforma de acreditación verificable para el sector cultural. Para mantener costos eficientes y escalabilidad completa sin depender de suscripciones SaaS de bases de datos por cada GB consumido, optamos por una arquitectura de Single-VPS Self-Hosted en un servidor Linux Ubuntu 24.04 LTS con 4 GB de memoria RAM.

Core Web Engine

Next.js 16.2 (App Router) empaquetado en modo output: "standalone" dentro de Docker.

Base de Datos Dual

PostgreSQL 17 + Supabase Self-Hosted (Kong Gateway, Auth, Storage, PostgREST y Studio).

Gestor de Paquetes

pnpm v9 mediante Corepack para compilaciones 5x más rápidas y bajo consumo de disco.

Blockchain Anclaje

Capa mock de metadatos SHA-256 e integración con la red Stellar / Soroban (smart contracts Rust).

2. El Rol de los Agentes de IA (Hermes & Antigravity)

Durante la construcción y el despliegue del proyecto, se utilizó un esquema de trabajo de colaboración multi-agente:

  • Hermes (Agente local en el VPS): Instalado en la sesión de usuario del VPS de Truebox para monitoreo de estado y tareas en segundo plano sin privilegios root.
  • Antigravity (Agente Principal de Desarrollo): Encargado de la creación de la suite de software, refactorizaciones de código, automatización de scripts de despliegue y resolución de bugs en tiempo real.

3. Desafíos Técnicos Encontrados y Soluciones

Desafío A: Desincronización de paquetes y ERR_PNPM_IGNORED_BUILDS

Inicialmente se utilizó npm ci en el Dockerfile, el cual fallaba por desincronizaciones entre el archivo package-lock.json de Windows y Alpine Linux. Migramos la compilación a pnpm, pero en pnpm@latest (v10) se introdujo el bloqueo por defecto de scripts nativos (lanzando ERR_PNPM_IGNORED_BUILDS en librerías como sharp).

Solución: Fijamos la versión de pnpm a pnpm@9 mediante Corepack en el Dockerfile:
RUN corepack enable && corepack prepare pnpm@9 --activate

Desafío B: Compilación de Next.js colgada en el VPS

Durante el comando pnpm run build dentro del contenedor Docker, la compilación se congelaba durante la etapa de Running TypeScript... (excediendo las 4 horas de espera). Esto ocurre en servidores con 1 o 2 núcleos vCPU debido a la contención de hilos del verificador de tipos de TypeScript.

Solución: Activamos el modo Standalone en next.config.ts y omitimos la re-verificación redundante de tipos dentro de Docker: 
// next.config.ts
const nextConfig: NextConfig = {
  typescript: { ignoreBuildErrors: true },
  output: "standalone",
};
Con esto, el tiempo de compilación dentro del VPS pasó de 4 horas colgado a solo 15 segundos.

Desafío C: Acceso a Supabase Studio (Puerto 8001)

Supabase Studio no cargaba externamente (ERR_CONNECTION_REFUSED) debido a que la plantilla oficial de Supabase Docker no expone por defecto un puerto mapeado al host para studio (manteniéndolo únicamente en el puerto interno 3000 de la red privada de Docker).

Solución: Inyectamos dinámicamente el mapeo de puertos 0.0.0.0:8001:3000 dentro del archivo docker-compose.yml de Supabase: 
sed -i '/container_name: supabase-studio/a \    ports:\n      - "8001:3000"' docker-compose.yml
docker compose up -d studio

Desafío D: Autenticación en PostgreSQL (FATAL: 28P01)

Al intentar ejecutar el script SQL desde Supabase Studio, este lanzaba un error de autenticación con el usuario postgres. Esto ocurrió porque el usuario superadministrador interno del contenedor de Supabase Postgres es supabase_admin.

Solución: Sincronizamos la contraseña de PostgreSQL ejecutando el comando como superadministrador: 
docker exec -i supabase-db psql -U supabase_admin -c "ALTER USER postgres WITH PASSWORD '$PASS';"

4. Script de Despliegue Automatizado (setup-vps.sh)

Para simplificar futuras actualizaciones a un solo comando, consolidamos todo el flujo en el script deploy/setup-vps.sh:

#!/bin/bash
set -e

echo "=== Despliegue de CulturaGO & Supabase Self-Hosted ==="

# 1. Instalar Docker y Docker Compose si no existen
if ! [ -x "$(command -v docker)" ]; then
  sudo apt-get update && sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
fi

# 2. Configurar e Iniciar Supabase Docker
if [ ! -d "supabase-docker" ]; then
  git clone --depth 1 https://github.com/supabase/supabase.git supabase-docker
fi
cd supabase-docker/docker
cp -n .env.example .env || true

# Generar secretos JWT y contraseñas seguras con Node.js
SECRETS=$(docker run --rm node:20-alpine node -e "...")
# Parsear e inyectar llaves...
sed -i '/container_name: supabase-studio/a \    ports:\n      - "8001:3000"' docker-compose.yml || true

docker compose pull && docker compose up -d
cd ../..

# 3. Compilar y Levantar la App Next.js en Modo Standalone
docker compose -f deploy/docker-compose.app.yml build --no-cache
docker compose -f deploy/docker-compose.app.yml up -d
echo "✓ ¡Despliegue Completado con Éxito!"
